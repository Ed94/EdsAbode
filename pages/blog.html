Spark Dev: IWVR

December 3nd, 2017

Back in September 5th, 2017, the final rosters for a new program from Startup FIU called Spark Dev had been given out via email. It was the pilot semester and wouldm run for 9 weeks. In it teams would work on projects with the focus being to learn from it them instead of making a finished product. I had signed up the last day of registration in hopes of getting in. There were four divisons to choose from: Artifical Intelligence, Simulation, Virutal Reality, and Web Development. I had signed up for the VR group and on the results of the roster my name was shown. The follow up email gave us the project conecpt we would be working on. A infinite virtual whiteboard. The project's source code will be hosted on github throughtout the entire program and will be avalible open-source to everyone. After the first general meeting we went on to figure out meeting schedules and settled at the beginning to do it every Friday.

First few weeks were just getting things settled, going over concepts for features we would want, and getting the some sort of struture down for the program. Back then we were divided into four groups: Programming, Modeling, Networking, and UI. Then got the github going before hurricane Irma came along and delayed things. Then I got the first unity started and a week after started we started to work on getting something started in scene. First thing was to get the line to draw on the board.

We were able to get the line to draw 2D on the board, and got an inital panning locomotion to work so the user could move around the board. The color was stuck to a pinkish purple due to not being able to set color properly. Initally the controller itself was used to draw the line. Next we dropped panning and worked with a smaller board and started to add teleporting. This allowed the user could move around the space, once the locomotion was in place we started to give the board proper collisions and creating the marker to draw with instead of the controller. The marker collison gave us quite a bit of trouble initally due to us using a kinematic approach to collision, that was within the hand script provided by steam vr. We were able to get it to work, however we had more trouble when the board was not in perfect orientation. When the marker attempted to draw the position of the line renders were all over the place and deformed. Through trial and error we were able to find the proper constants for the line render scaling and the proper transform space they should position with along with the rotational tranformations that needed to be done so that the render was perfect no matter what the boards orientation was. This was are second most time consuming problem.

Next was getting board and marker to have proper physics, and allowing the creation of boards. For the most part this went pretty smoothly except with keeping the board still when the marker hit it. We fixed this tempoarily by using fixed joints but the are not a viable solution since it can become a performance killer if boards have bad interactions. Last few things were getting the eraser to work along with color and size selection for the marker and board resizing. The The eraser gave us quite a bit of trouble when it came to detecting collsion with the lines. Trying to give them unity's line renders did not work. What we settled with was so that every vertex of the line after rendering getting a small box collider. Color and Marker size were implemented with minor difficulty. Board resizing was thrown in with relative ease however its implementation was poor and is slightly difficult for the user to use the resize controls.

In the end we ended up with fully functional board, marker, and eraser with ability to create and resize boards, along with marker color and size customization. We unfortunately could not get the networking side implemented for cooperation. We had last half our team by the end of the program and had to allocate to much time to getting Unity's physics and collision to work properly with our VR interactions. In the future we plan to port the project over to Unreal Engine and continue from where we left off in the coming semester with the second iteration of Spark Dev.

Immersive Tech Club (FIU): The First Game Day and Workshop

November 17th, 2017

Yesterday the Immersive Tech Club held their first Virtual Reality Workshop in coollaboration with WICS(Women in Computer Science). The workshop consisted of going over how to use Unity and the SteamVR plugin to create VR content. Specifically what are the core items used in a Unity scene, and how to create various interactions possible with SteamVR with various objects. This was done by first making sure everyone had Unity and SteamVR installed. Then those attending downloaded a prestructured project from a github repo we had prepared beforehand to follow on with during the workshop. The project essentially had a preset scene with objects that would have all the SteamVR related prefabs and components needed. These resources would then be addeed and configured to the scene as the workshop went over all its material.

The workshop was a great sucess, we had more attendees than there were seats in the room. The majority were able to follow along and understand the material. We has a select few who had trouble getting 2D debug to work, this had to do with conflict with SteamVR not being able to detect a headset. Normally when this occurs its supposed to startup 2D debug in unity however in this case it still attempted to run the Unity scene instance with the headset. 2D debug essentially allows you to interact in the scene with what are called fallback objects in the same way anyone would if there was an actual headset. Without it nothing added to the scene could be tested without an actual headest. Unfortunately for those we were not able to solve the issue until nearly after the workshop was over. But we were able to at least give them a quick summary of it after.

The next day was the club's first self hosted event: The VR Game Day. Room 116 of PG6 was reserved from 7 to 10pm. There was 4 VR stations present, each had a different set of games. Mine was the racing simulation, Hazar's: shooting and archery, Adam's Flying and space simulations, and Ciana's Playstation VR. While everyone waited for their time on a VR setup, there are raspberry pi setups with a set of games from a specifed console. These included the nintendo 64, super nintendo, and sega genesis. We also had rocket leauge and a nintendo switch. To help with the atmosphere we had music from spotify playing the background. The event was a sucess, people really got into every setup and enjoyed them. A good percentage of those trying out every setup had never used VR and it was great to watch their reactions. Our side games kept people busy if they were not watching others in VR however the raspherry Pi's did not seem to get as much attention. Most likely from the fact that the super nintendo setup seemed far from where everyone tended to gather and that some had not had thier peripherals properly setup. This was negligable as the main focus was the upkeep of the four VR stations.

Overall the events were a great opening to the founding of the Immersive tech club, and really started to those that participated exposed the world of VR. I had a ton of fun hosting the racing station and hope to do it again some time in the future.

Melee Remastered Top 8 Complete Released!

July 29th, 2017

I first got into the smash scene through a modification of Super Smash Bros. Brawl called Project M. The mods purpose was to make brawl like its previous interation: Super Smash Bros. Melee. The game features a major rebalance to the cast; along with adding back a bunch of the mechanical features that made melee the competitive fast pace technical frenzy, which continues to be played as an esport to this day. Project M had a great modding scene, and I decided to use the Post-Processing capabilites of a custom distribution of Dolphin, called Ishiiruka, to configure my own shader for the game. The shader was suprisingly well recieved by the community as well.

While Project M dwindled in popularity from legal hurdles, Melee is still strong and enjoy's a thriving community, I've also grown to like its feel over Project M, so I started to mod it and play it more. The first thing I did was get the 20XX hackpack and attempted to make a shader for Melee as well. Ishiiruka-Dolphin was able to add Tone Mapping, SSAO, SSGI, Bloom, Pixel Vibrance, Phong Lighting, Auto-Generated Bump maps, and more without completely destorying the peformance of something that was already trying to emulate the hardware of the gamecube.

The result of the shader was alright, but seemed over done and the characters looked to washed out. The reason was to try to fix the incorrect colors on many characters. The color balance on many textures was incorrect as well as other things, leading me to over compensate for it in the shader. So I tried my hand at correcting the textures for the most popular character in the game, Fox. Many of his textures for his clothing needed to be color corrected such as his jacket which was tinted greenish beige, his fur was also over saturated with yellow. With the color correction I also did small touch ups, and redid the tone curve for each texture. Due to the amount of textures involved along with creating custom Character portraits and corrected stock icons, to finish the character in all his skins took around 5 hours(Longest since it was the first). The result of it was unbelivable. The character looked much cleaner and fresh; similar to a restoration of a vintage vehicle, without the use of any shaders.

Before

After

With the satifsying result motivating me, I decided to continue to remastering various textures. The project was named Melee Remastered and so far focused on improving the shader and textures of the game. Curernt focus was on getting the characters and legal stages for competitive play done. The characters were done in order of the 2015 tier list, and after over a year, the first major release of this project has been released as "Top 8 Complete". The characters completed are Fox, Falco, Marth, Shiek, Jigglypuff, Peach, Captain Falcon, and Ice Climbers. To be complete, all their skins needed to be finished, along with new Character Select Portraits for the selection menu, and remastered stock icons. As for the stages, they have had all their textures done, however I have plans in the future to adjust the stage lighting without using shader by hacking the file. The shader is also completey different with the lastest version looking almost as good(if not better) than the one I originally made for Project M.

Since the lastest release, the project has garnered a positive reponse along with plenty of feedback. I was suprised just how much you can learn from somthing like a modding project past just using an image editor and processs to adjust texture. My original intent with this project was to become proficient in Photoshop and Gimp. However doing this project allowed me to further improve my ability to stream and record with Open Broadcaster Studio, furthering me into the world of color calibrating displays, using batch files in windows, and some PowerPC assembly. Plus it gave me and a better understanding of marketing something properly, obtaining feedback, and properly managing such a thing. Its suprising how much is involved with just producing such a mod for a game. To an extent, it can be almost as much work as making one.

Knowing that those in the community want to see more, I will continue the project at least until all the characters are completed, and the stage lighting has been adjusted.

Mango Hacks 2017

February 24-26, 2017

Went to my first hackathon. It was similar to the Global Game Jam. We were given roughly 60 hours to make something. This was probably the best thing that happend in my spring semester at FIU. There was a bunch of energy, activity, and plenty of people to meet and have fun with.

Since I had never gone to one of these before I brought evertyhing I had that would be useful, this included, my laptop, desktop, a bunch of gamepads, bluetooth speaker, headphones, and an HTC Vive. At the entrance most of the sponsors had booths and discussed with us. State Farm asked us to fill out a survey, and IBM was giving out a Rasphberry PI 3. I picked one of those up quickly as the supply was going to run dry soon.

After registering myself, I went to look at the space the hackathon had reserved. It was essentially the entire first floor of PG6 at FIU. The louge area kept its same purpose, while the rest of the lecture rooms were used as spaces for formed teams to work on their individual projects. The only exception being the small lecture room used as a quiet area and some of the computer labs used for workshops they would have throughout the hackathon. After around 20 minutes I found three people looking for a group and was able to team up with them. We then setup in room 112 as it had the most empty space and we would need it for VR.

After the opening ceremony, we went on to brainstorm what we were going to do with VR. Initial plans were to spew out fractals from the vive's controllers be soon realized the fractals formula's may be to complex for the amount of time we had. Then we decided to do a sort of shooting gallery with the player and enemies flying around in space. From there we devised our roles on the project. I was in charge of figuring out a proper way of doing locomotion to fly through space. while retianing control. We unanimously decided to use unity to work on VR, as we were exposed to it earlier at the Unity workshop from UPE and it was easy to implement VR with.

Near the end of the night, I met a guy named Matthew who asked if he could share the VR space for his own project he was working on since I already had one setup. He lent us his tripods aswell so we didn't need to keep our bootleg setup for keeping the base stations up. His project was on getting a sort of earth bending expereince working, where the player could manipulate brown blocks. He with another guy named Haczar, I also found out they were involved in OpenHid, and that the HTC vive they brought was from the lab.

Nearly half way through I had figured out a good locomotion VR involving the player producing a ray from the controller into the direction in virual space they wanted to travel. After the ray is placed out, all they need to do is press the trigger and the motion's velcoity gets translated to acceleration toward the direction of their choosing. To change their position all they would need to do is shoot out a new ray in the direction of their choosing. While I had the concept down, getting it to work in unity proved to be a challenge as I was not familiar with its libraries or objects used to handle VR. While the ray seemed to working, amd I was getting a velocity from the controllers when pressing the trigger I could not get it to change cammera's position in the virutal space. My teammates had also been fatigued after getting the weapon projectile properly working, and attempting to build a proper level and enemy AI. After saturday night they were not willing to continue in the event. I decided to keep going over night to see if I could at least get the locomotion working. Unfortunately even with the aid of various addons that make VR easier and getting quite close by using some prewritten code from example scenes, I was not able to get the locomotion working before the end of the event.

In the end the event was still a blast and met a ton of great people! I'll be finishing the locomotion some time in the future as well.

Global Game Jam @ FIU

Januaray 20-22, 2017

During the first meeting at UPE that I attended, they discussed a sort of 'hackathon' for video games called a game jam." As someone already making a game in their free time I was naturally intrested to I signed myself up for the event. Essentially we had a bit less than 72 hours to make a video game based on a theme announced during the opening ceremony for the event.

I was able to quickly team up with a group. We at first decided to do a top down game where a character navigates using echo location, avoiding enemies as they go. We then scrapped that half way through and went with 3D horror where red radar waves scan through the entire maps's models to give player's a sense of where they are while they run away from various enemies they encounter as they traverse through the map. If they are able to avoid all the enemies and get through they win. If they get touched by a single enemy they lose.

We were able to finish the game on time and during the showcase, the game became quite popular for its natural suspense created by the lighting effect and the lack of awareness of enemy position. Other teams also had great games, one created a game where the player was a monkey looking for a key, while making a mess at a company office. Another was a bat flying through a cave with similar wave lights to our game, a platformer where the enviornment moves based on the music being played. During awards, people voted ours as best overall experience and we won a game making book, Level Up Game Design!
